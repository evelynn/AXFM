# AXFM 정식 서비스 전환 보완 계획 (2026-07-03)

> 입력: 6개 관점 독립 연구원 검토(발견 48건 — `docs/reviews/2026-07-03-service-review.md` 전문).
> 목적: v0.2.0("파일럿 착수 가능한 프로토타입")을 **수십 명이 상시 사용하는 정식 사내 서비스**로 승격하는 경로 정의.

## 종합 판정 (6트랙 수렴)

- **기술 코어는 견고하다** — 결정적 스크립트, 원자적 레지스트리, 크로스 스택 E2E, 행동형 한국어 오류 메시지는
  전 트랙이 강점으로 수렴. 아키텍처 재설계는 불필요.
- **남은 무게중심은 코드가 아니라 ① 문서-실환경 정합(PS 5.1) ② 규약의 마지막 모순(보내기 방향) ③ 운영 체계**
  (배포 채널·업데이트 전파·지원 주체·계측)다.
- 정식 서비스 선언은 **파일럿 데이터와 운영 요건이 갖춰진 뒤** — 아래 GA 게이트로 판정한다.

## Phase 3 — 정식 서비스 선언(GA) 및 확산

- **GA 게이트 (전부 충족 시 선언)**: ① 파일럿 정량 기준 통과 ② 운영 요건(담당자·채널·릴리스 절차·CI) 가동
  ③ 보안 패키지(데이터 규칙·공급망 통제·위협 모델) 승인 ④ 게이트 전체 녹색(신설 포함) ⑤ CHANGELOG·태그 운영 중
- 확산: 1팀 → 부서 → 전사 단계별 진입 기준, 사내 git 조직/토픽 기반 팀 카탈로그(주간 스캔), id owner-prefix 기본 강제
- 후속 로드맵: capability 스키마 버전(무음 파손 감지), major 이행 규칙(N-1 허용), `/axfm-adopt`(오너 결정대로 후순위), 사람 간 데이터 공유

## 오너 결정 필요 (5건 — 이것 없이는 해당 항목 착수 불가)

1. **보내기 모델** (0-2): (a) readFrom 단일화(코드 0) vs **(b) writeTo+inbox(권고)**
2. **canonical 사내 git 저장소 주소** (1-1) — 지금은 원격이 없어 배포 채널이 실존하지 않음
3. **운영 담당자와 지원 채널** (1-8) — 주/부 담당, 사내 메신저 방 또는 이슈 트래커
4. **파일럿 표본** (1-7) — 참가 인원·팀, 비개발자 비율, 일정
5. **외부 스킬 정책** (2-5) — 사내 allowlist 전환 여부

## 진행 방식 제안

Phase 0은 전부 코드·문서 수정이라 **오너 결정 1번(보내기 모델)만 정해지면 즉시 착수 가능**하다.
Phase 1의 스크립트·게이트(1-3~1-6)도 결정 없이 진행 가능하며, 1-1·1-2·1-8은 오너 입력(주소·담당자)이 필요하다.
