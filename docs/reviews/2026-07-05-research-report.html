v0.2.0은 "잘 만들어진 프로토타입 + 미검증 약속"이었다. v0.5.0은 "약속한 것은 게이트가 지키는 상태"다. 파일럿 착수를 막는 기술 항목은 0건이며, 남은 blocker 3건은 전부 오너 결정 대기(운영 담당자·지원 채널, 현황판 승인, 파일럿 표본)다.

1. 연구·업그레이드 경과

  • 7/3재검토로 결함 발견(PS5.1 비호환·데모 미컴파일·크로스 스택 파손) → v0.2.0: 설치 가능 구조 완성, 폴더 스코프 확정, GitHub 공개(evelynn/AXFM)
  • 7/36개 관점 독립 연구원 검토(UX·아키텍처·보안·운영·제품·품질) — 발견 48건(blocker 15·major 24·minor 9), 정식 서비스 전환 계획 수립
  • 7/4v0.3.0: Phase 0 전 항목 — 보내기 모델 확정("보내기도 받아가기"), 스킬 9→8종, 이름 규칙 엄격화, 레지스트리 견고화, 문서 게이트
  • 7/5v0.4.0: 게이트 8종(스킬 린트·스크립트 게이트), publish-check(유출 자동 점검)·check-env, lockfile(의존성 57개 고정)+release-check, 릴리스 절차·정량 파일럿 게이트

버전업은 매회 실제 사용자 경로(marketplace update → plugin update)로 배포 실증: 0.2.0→0.3.0→0.4.0→0.5.0

판독

  • 품질·보안·UX 상승은 이틀의 투자가 게이트·유출 방지·명령 호환에 집중된 결과 — 사용자가 첫 10분에 만나는 실패 표면이 구조적으로 줄었다.
  • 운영(38점)이 최저 — 남은 것이 CI·계측·담당자 지정 같은 "운영 체계" 항목이라 코드보다 결정이 필요한 영역. 다음 투자처.
  • 발견 48건 처리: 해소 16 · 부분 14 · 잔여 18 — 잔여의 대부분은 (a) 오너 결정 (b) 확산 규모(30~50개)에서만 필요한 항목 (c) 사람 실측(파일럿).

4. 토큰 효율 연구 (설계 4원칙 "알뜰"의 근거)

일반 사용자의 요금제는 Pro가 기본이라는 전제에서, 스킬의 비용 구조를 측정하고 절약 원칙을 수립했다.

비용 구조 — 두 종류

상시(always-on) — 활성 폴더의 모든 세션마다 스킬 설명이 컨텍스트에 실림.
실측 합계 ~1,132 tok/세션 (9종 1,266 → 8종 정리로 -134). 관리 목표 1,500 미만, 릴리스마다 재측정.
호출 시(on-invoke) — 발동한 스킬의 본문 + 스킬이 읽는 파일들.
핵심 통찰: 진짜 비용은 본문이 아니라 "LLM이 하는 일" — 절약의 중심은 문장 압축이 아니라 구조다.

5. 결론과 다음 단계

  1. 기술 준비는 파일럿 기준 완료 — 설치→생성→실행→연동→공유 전 구간이 자동 게이트로 보증되고, 버전업 배포 경로가 4회 실증됨.
  2. 다음 병목은 결정과 실측: ① 운영 담당자·지원 채널 지정 ② 파일럿 표본·일정 확정(정량 GA 게이트: 80% 무개입 완주·중위 ≤10분) ③ 현황판 솔루션(overview() 기반) 제작 여부.
  3. 확산 규모 대비 백로그(아키텍처·운영 잔여): interface 파서 골든 케이스, sync 백업/롤백, 일괄 동기화, CI, 위협 모델 문서 — 파일럿 병행 진행 가능.
