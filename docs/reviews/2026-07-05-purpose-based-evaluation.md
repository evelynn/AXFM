# 목적 기반 재평가 (2026-07-05, v0.5.0)

> 잣대: purpose.md 의 정체성 — **"완성형 제품이 아니라, 여러 명이 바이브 코딩할 때 연결과 디자인 통일을
> 위한 기본 프레임워크. 완성도의 기준은 기능의 많음이 아니라 틀의 견고함."**
> 방법: 정체성에서 평가 축 6개를 도출하고, 근거(코드·게이트·실증)로 채점. 기존 발견 중 "완성형 제품의
> 요건"에 해당하는 것은 **목적 밖(로드맵)** 으로 명시 분리. 이전 점수표(정식 서비스 기준 50점)와의 차이는
> 상태 변화가 아니라 **기준 변화 + v0.5.0 반영**이다.

| ② 디자인 통일 틀 | 20% | 정체성의 두 번째 존재 이유 |

| ⑥ 비개발자 접근성·알뜰 | 10% | 과제 ①② + 4원칙 |

## 축별 채점

### ① 연결 틀 — **85점**
- ✅ 규약 모순 0: 보내기 모델 확정("보내기도 받아가기", 다중 발신 충돌 없음), v1 잔재 제거 후 게이트로 봉인
- ✅ 크로스 스택(웹앱↔스크립트) 실데이터 왕복이 상시 게이트 — 봉투·이름·발신자 검증 대칭 구현
- ✅ 계약 체계: interface.md + manifest + 표준 봉투 + §8 제3스택 체크리스트 / ✅ overview() 현황 데이터 소스
- ⚠️ 잔여: interface.md 파서 골든 케이스(따옴표·중첩 엣지 — 연결 틀의 마지막 구멍), payload 스키마 버전(확산 규모용)

### ② 디자인 통일 틀 — **80점**
- ✅ 표준 v1.0 오너 확정, 단일 소스(design/) → 전 솔루션 수출, 교체 시 코드 변경 0 (베이스 프레임워크의 핵심 요건)
- ✅ "토큰만 사용" 규칙이 스킬(feature·qa)에 내장, 수출 검증 게이트 상시
- ⚠️ 잔여: 폰트(Pretendard/D2Coding) 미번들 — 미설치 PC 는 시스템 폰트로 대체, 로컬 오버라이드 통제 규칙(W-D4)은 계획만

### ⑤ 틀의 견고함 — **85점**
- ✅ 자동 게이트 8종 + release-check: 규약·스킬 지시문·스크립트 행동·버전 3축·문서 호환·한글 경로까지 보증
- ✅ 업그레이드 경로 4회 실증(0.2→0.5), lockfile 재현 빌드, 릴리스·롤백 절차 문서
- ✅ 게이트가 실결함을 실제로 검출한 이력(스킬 린트 첫 실행, publish-check 비밀 주입 거부) — 장식이 아님
- ⚠️ 잔여: CI 부재(게이트 실행이 사람 관행에 의존), 레지스트리 동시 쓰기 스트레스 미검증

### ③ 다인 사용 성립 — **75점**
- ✅ 폴더 스코프(전역 오염 0) 실측, GitHub 공개 배포 채널 + 태그, publish-check 유출 자동 차단

### ④ 고쳐 쓰기 용이성 — **70점**

## 목적 밖으로 재분류된 항목 (완성형 제품·전사 서비스의 요건 — 로드맵 보관)

- 전사 확산 플레이북·단계별 진입 게이트 / KPI 자동 집계·사용 현황 계측 / 90분 실습 커리큘럼
- 사람 간 원격 데이터 공유 / 팀 카탈로그 자동화 / 외부 스킬 allowlist 운영 체계
- 이들이 "잔여 결함"이 아니라 **"기본 프레임워크 위에 조직이 얹을 것"**이라는 게 재분류의 의미다.
  단, 운영 담당자·지원 채널 지정은 목적 안(다인 공유의 최소 요건)으로 유지한다.
