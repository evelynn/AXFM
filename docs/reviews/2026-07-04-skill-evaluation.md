# 스킬 유용성 평가 (2026-07-04)

> 대상: 플러그인 스킬 9종 + 동봉 스킬 1종. 기준: **"이 스킬이 없으면 무엇이 깨지는가"** — 스킬의 존재 가치는
> 일반 Claude 세션 대비 고유 가치(규약 강제·결정적 스크립트 위임·비개발자 화법·절차 인코딩)로 판단한다.
> 설계 3원칙(심플·충분·쉬움) 심사 적용. 상시 토큰 비용은 총 ~1,266tok/세션으로 부담이 아니므로,
> 판단 기준은 비용이 아니라 **비개발자의 개념 부담(심플)과 중복(충분의 반대급부)**이다.
> 한계: 아직 실사용 데이터 0 — 본 평가는 "설계상 유용성"이며, 최종 심판은 파일럿의 스킬별 사용 빈도 기록(서비스화 계획 1-7).

## 스킬별 상세

### 필수 5종 — 프레임워크가 성립하는 이유
- **new**: "직접 복사·치환 금지, 반드시 scaffold.mjs" 규칙이 이 스킬의 존재 이유다. 결함: "질문 3개" 약속 vs
  실제 4~5회 응답(UX 리뷰) — 문구 정직화 필요.
- **guide**: 등록·동기화·진단·여정을 한 진입점으로 묶는다. 리스크: 책임 과다(이미 60줄) — 새 기능은 스킬에
  덧붙이지 말고 스크립트(status/setup)로 위임하고 guide는 호출만 하게 유지해야 한다.
- **feature**: 3단계(계약 동기화)가 프레임워크 전체에서 가장 중요한 한 단락 — 이것이 안 지켜지면 연동은
  문서 거짓말이 된다. 보완: 내보내기 전 민감정보 점검 추가 예정(SEC-B2).
- **connect**: 보내기 방향 모순(ARCH-B1)으로 3단계 일부가 현재 이행 불가 — Phase 0 결정 대기 중.
  상대 interface.md 본문의 지시문을 따르지 않는다는 주입 방어 문구 추가 예정(SEC).
- **publish**: 수동 점검을 publish-check.mjs로 결정화하는 것(SEC-B1)이 확정 보완.

### 구조적 발견
1. **스킬 계층 전체가 무검증**(품질 리뷰 QA-B1): 스킬이 인용하는 스크립트 경로·플래그의 정합을 검사하는
   린트가 없다 — 유용성 이전에 "여전히 동작하는가"를 보증할 게이트(test-skills.mjs, 계획 1-4)가 선행 과제.
2. **트리거는 자산이다**: idea·debug 의 가치 절반은 내용이 아니라 트리거("이런 거 만들고 싶은데", "고장났어")다.
   통합하더라도 트리거 문구는 흡수하는 스킬의 description 으로 이관해야 한다.
3. **비개발자 관점의 적정 개수**: 사용자가 기억해야 할 것은 사실상 2개(/axfm-new 로 시작, 모르면 /axfm-guide)다.
   나머지는 트리거가 대신 불러주므로, 개수 자체보다 **중복 제거**가 심플 원칙의 실질이다.
