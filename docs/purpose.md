# AXFM 목적 선언 (2026-07-03 오너 확정, 2026-07-05 정체성 보강)

> AI(바이브 코딩)로 다양한 업무 솔루션을 만들고 있고 앞으로 더 만들 예정이다. 그 과정의 다섯 가지
> 어려움을 **쉽게 만들어주는 플러그인**이 AXFM 의 존재 이유다. 모든 설계·기능 결정은 이 다섯 과제로 정당화한다.

## 정체성 (2026-07-05 오너 확정)

AXFM 은 **완성형 제품이 아니다**. 회사·그룹에서 **여러 명이 바이브 코딩할 때 솔루션 간 연결(연동)과
디자인 통일을 위한 "기본 프레임워크(베이스)"**다. 이 정체성이 뜻하는 것:

- 제공하는 것은 **틀**이다 — 공통 규약(protocol)·공통 디자인(design)·공통 함수(modules)·개발 가이드(skills).
  그 위에서 각자 만드는 솔루션의 내용에는 관여하지 않는다.
- **팀에 맞게 고쳐 쓰는 것이 전제**다 — 디자인 교체, 규약 확장, 동봉 스킬 추가가 정상적인 사용법이다.
- 따라서 "완성도" 평가의 기준은 기능의 많음이 아니라 **틀의 견고함**(규약 모순 없음·게이트 보증·업그레이드 경로)이다.

## 설계 4원칙 (오너 확정 — 사내 표준 프레임워크의 심사 기준)

모든 기능·폴더·스킬·문서는 추가·변경 전에 아래 원칙으로 심사한다:

1. **심플** — 사용자가 알아야 할 개념을 최소로. 폴더·스킬·명령이 늘 때마다 "빼도 되는가"를 먼저 묻는다.
   빈 폴더·장식 구조·중복 기능 금지.
2. **충분** — 만들기→기능→연동→공유의 전 여정을 스킬이 빈틈없이 커버한다. 부족한 것은 채우되,
   이미 있는 것(예: /axfm-qa)을 복제하지 않는다.
3. **쉬움** — 기본 환경(Windows·PowerShell 5.1·비개발자)에서 복붙만으로 동작해야 한다.
   **사용자가 터미널(CLI)에 익숙하지 않을 수 있다** — 스킬은 명령을 시키기 전에 대신 실행을 제안한다.
   위험하거나 어려운 단계는 사람이 아니라 결정적 스크립트가 대신한다.
4. **알뜰 (토큰 절약, 2026-07-05 추가)** — 일반 사용자는 Pro 요금제가 기본이다. 스킬은 필요한 파일만
   읽고 짧게 답하며, 반복 집계·검사는 LLM 이 아니라 스크립트·공통 함수가 수행한다. 상시(세션마다)
   토큰 비용은 측정·관리한다 — 연구·측정: `docs/design/token-efficiency.md`.

## 5대 과제와 현재 커버리지 (v0.2.0 기준)

### 과제 1. 일반 사용자가 개발을 배우기엔 아직 배울 게 많다
- **커버**: `/axfm-guide`(상태 진단 → 지금 할 일 딱 하나), `/axfm-idea`(막연한 아이디어 → 요구사항),
  5단계 여정 게이지, 생성물마다 CLAUDE.md(비개발자 배려 규칙 내장), 전문용어 한 줄 설명 원칙.
- **남은 것**: 파일럿 M3 실측(사람의 10분 스톱워치)으로 실제 초보자에게 통하는지 확인.

### 과제 2. 환경 구성과 스킬 세팅이 어렵다
- **커버**: 설치 2줄(폴더 스코프 — 해당 폴더에만 적용, 전역 오염 없음), `/axfm-new`가 환경 점검(Node/Python)
  포함 결정적 생성, 추천 도구 큐레이션(guide 가 recommended-skills.md 참조 — v0.3에서 toolbox 스킬 흡수), troubleshooting 증상표, npm 막힘 진단.
- **남은 것**: 관리자 1회 세팅(마켓플레이스 등록, Node/Python 설치)의 자동화·체크 스크립트는 없음
  (현재는 pilot-onboarding.md 의 수동 체크리스트).

### 과제 3. 솔루션 간 인터페이스
- **오너 결정 (2026-07-03)**: **이 플러그인으로 만든 솔루션끼리의 연동이 우선. 기존(비-AXFM) 솔루션 편입은 후순위.**
- **커버(우선 범위 — 완료)**: 신규 생성 솔루션끼리는 표준 통로(interface.md + 공통 함수 + 스냅샷 봉투)로
  연동 성립 — python↔python·nextjs↔python 실데이터 왕복이 자동 검증으로 상시 확인됨.
- **후순위(로드맵)**: `/axfm-adopt` — 기존 프로젝트에 질문 몇 개 → `axfm.json`·`interface.md` 생성 →
  `sync-modules.mjs` 모듈 주입 → `register.mjs` 등록. 기존 스크립트 전부 재사용이라 착수 시 비용 낮음.

### 과제 4. 시스템마다 디자인이 제각각이다
- **커버**: 플러그인 중앙 `design/*.md`(DESIGN.md 규격)가 단일 진실 — 모든 UI 작업 전 필독 규칙,
  토큰만 사용(임의 색·간격 금지), `export-design.mjs`가 CSS 변수로 결정적 수출, 드리프트 감지·재생성.
- **오너 결정 (2026-07-03)**: **현재 design/DESIGN.md 를 표준으로 확정** (placeholder 상태 종료 — v1.0).
  추후 브랜드 변경은 파일 교체/오버라이드로 (코드 변경 0).
- **남은 것**: 폰트(Pretendard/D2Coding) 미번들 — 미설치 PC 는 시스템 폰트로 대체됨.

### 과제 5. 시스템 개발에 대한 관리가 안 될 수 있다
- **커버**: 로컬 레지스트리(이 PC의 솔루션 목록·경로·버전), 솔루션별 진행 여정(progress), 모듈 버전
  드리프트 감지, `/axfm-publish` 공유 체크리스트 + publish-check 자동 유출 점검.
- **커버 (2026-07-05 오너 지시로 추가)**: 공통 함수 **`overview()`** — 이 PC 전체 솔루션의 생존·모듈 버전·
  스냅샷 신선도·provides/accepts 를 한 번에 반환. **종합 현황판 솔루션을 AXFM 스스로 만들 수 있는 데이터 소스**
  (집계는 코드가, 화면은 앱이 — 토큰 절약).
- **로드맵**: `overview()` 기반 **시스템 일괄 모니터링 솔루션**(전 솔루션 상태를 한 화면에), 팀 차원
  카탈로그(사내 git 조직/토픽 규약).
