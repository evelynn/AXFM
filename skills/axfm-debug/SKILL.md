# 문제 해결 가이드

디버깅 방식은 평소대로: 증상(무엇을 했을 때 무엇이 보였나 + 에러 메시지 전문)을 먼저 정확히 받고,
가설 하나 → 확인 하나 → 결과 보고로 좁힌다(추측으로 여러 곳 동시 변경·막연한 재시도 금지). 비개발자일 수 있으니 전문용어는 한 줄로 푼다.

이 스킬의 고유값은 아래 **AXFM 특화 원인 순서표** 하나다 — 일반 원인보다 이걸 먼저 의심한다.

## AXFM 흔한 원인부터 (순서대로 확인)
1. **환경**: `node "${CLAUDE_PLUGIN_ROOT}/scripts/check-env.mjs"` (웹앱이면 `--web`) — 실패 항목 안내를 그대로 전달.
2. **npm 막힘**(사내망): `npm install` 실패 → `docs/troubleshooting.md` + 사내 레지스트리 안내.
3. **실행 위치**: 프로젝트 폴더(axfm.json 있는 곳)에서 실행 중인가?
4. **PowerShell 실행정책**: `.ps1` 이 막히면 `.\start.cmd` 사용 (정책 우회는 최후 수단 — 관리자 확인 후).
5. **연동 문제**: 상대가 데이터를 내보냈나(`.axfm/data/<name>.json` 존재)? 레지스트리 경로 정확? 규약 버전 일치?
6. **모듈/스킬 버전**: `/axfm-guide` 로 드리프트 확인.

## 마무리
- 고친 뒤 원래 증상이 사라졌는지 사용자가 눈으로 확인하게 한다.
- 같은 문제가 반복되면 `docs/troubleshooting.md` 에 추가를 제안한다.
