# AXFM

바이브 코딩(AI에게 시켜서 만드는 개발)을 위한 **Claude Code 플러그인 프레임워크**입니다.
개발이 처음이어도 괜찮습니다. `/axfm-new` 한 줄이면 동작하는 솔루션 뼈대가 생기고,
`/axfm-guide`가 지금 할 일을 하나씩 짚어줍니다.

> **이 프로젝트의 성격**: 완성형 제품이 아니라, 회사·그룹에서 **여러 명이 바이브 코딩할 때
> 솔루션 간 연결(연동)과 디자인 통일을 위한 기본 프레임워크(베이스)**입니다.
> 각자 만든 도구가 따로 놀지 않도록 공통 규약·공통 디자인·공통 함수의 틀만 제공하며,
> 그 위의 솔루션은 각자 자유롭게 만듭니다. 팀에 맞게 고쳐 쓰는 것을 전제로 합니다
> (디자인 교체: `design/`, 규약: `docs/protocol.md`).

- **만들기**: `/axfm-new` 한 번으로 동작하는 솔루션 뼈대가 생깁니다.
- **연동하기**: AXFM으로 만든 솔루션끼리는 공통 규약으로 쉽게 데이터를 주고받습니다.
  (연동 범위는 **같은 PC** — 팀원 솔루션은 clone 받아 내 PC에서 실행하면 이웃이 됩니다. 실시간 서버 없음.)
- **배우기**: `/axfm-guide`가 지금 할 일을 딱 하나씩 알려줍니다.

## 현장의 문제점 (AXFM이 푸는 5가지 과제)

AI로 각자 업무 도구를 만들기 시작하면, 도구가 늘어날수록 다음 다섯 가지 어려움이 반복해서 나타납니다.
AXFM의 모든 설계·기능은 이 다섯 과제로 정당화됩니다.

| # | 현장의 문제점 | 방치하면 생기는 일 |
|---|---|---|
| 1 | **일반 사용자가 개발을 배우기엔 배울 게 너무 많다** | 시작 자체를 못 하거나, 첫 화면을 띄우기까지 며칠씩 헤맨다. |
| 2 | **환경 구성과 스킬 세팅이 어렵다** | Node/Python 설치, 마켓플레이스 등록, 실행 정책 등 진입 장벽에서 이탈한다. |
| 3 | **솔루션마다 인터페이스가 제각각이다** | A가 만든 데이터를 B가 못 읽는다. 연동할 때마다 매번 새로 규격을 협의해야 한다. |
| 4 | **시스템마다 디자인이 제각각이다** | 회사 도구들이 전부 다른 얼굴 — 브랜드 통일도, 사용자 학습도 안 된다. |
| 5 | **시스템 개발에 대한 관리가 안 된다** | 누가 무엇을 어디에 만들었는지, 뭘 주고받는지 아무도 전체를 모른다. |

**핵심 통증**: 개인이 AI로 도구 하나를 만드는 것은 이제 쉽다. 그러나 **여러 명이 만든 도구를
'연결'하고 '통일'하고 '관리'하는 것**은 여전히 어렵다 — AXFM은 바로 이 틈을 메우는 베이스다.

각 과제의 현재 커버리지와 로드맵은 [docs/purpose.md](docs/purpose.md)에 상세히 정리되어 있습니다.

## AI 활용방안

AXFM은 "AI가 코드를 대신 짜준다"에서 멈추지 않고, **AI가 실수하기 쉬운 지점을 결정적 스크립트로
막고, 사람이 판단할 지점만 AI가 대화로 돕도록** 역할을 나눕니다.

### 1) 바이브 코딩 전 여정을 스킬이 커버
자연어로 말하면 알맞은 스킬이 동작합니다 — 만들기 → 기능 → 연동 → 공유까지 빈틈없이.

| 스킬 | 언제 쓰나 | 하는 일 |
|---|---|---|
| `/axfm-new` | "이런 거 만들고 싶은데", "새 솔루션 만들어" | (막연하면) 아이디어 정리 → 질문 3개 + 확인 1번으로 동작하는 뼈대 생성 |
| `/axfm-guide` | "다음에 뭐 하지", "막혔어" | 상태 진단 후 **지금 할 일 딱 하나** 안내 |
| `/axfm-feature` | "이 기능 추가해줘" | 기능 추가 + 연동 계약(interface.md) 자동 갱신 |
| `/axfm-connect` | "저 솔루션이랑 연결" | 상대 문서를 읽고 연동 글루 코드를 생성 |
| `/axfm-debug` | "안 돼", "에러 나" | 초보자 언어로 원인 진단·수정 |
| `/axfm-publish` | "배포 전 점검", "팀에 공유" | 동작·정합 점검(QA) + 공유 준비 + 민감정보 유출 자동 점검 |

### 2) "위험·반복은 AI가 아니라 스크립트가" — 역할 분담
- **AI(대화)가 맡는 것**: 아이디어 구체화, 기능 설계, 연동 대상 판단, 오류 원인 추정.
- **결정적 스크립트가 맡는 것**: 생성(scaffold)·등록(register)·모듈 동기화(sync-modules)·
  디자인 수출(export-design)·유출 점검(publish-check). → **누가 언제 돌려도 같은 결과**, AI의 무작위성 배제.
- **집계·검사는 코드가**: 반복 집계(현황판 등)는 LLM이 매번 계산하지 않고 공통 함수 `overview()`가 수행 →
  정확도↑·토큰↓.

### 3) 비-CLI 사용자 배려
스킬은 명령을 시키기 **전에 "제가 대신 실행할까요?"**를 제안합니다. 터미널이 낯설어도
채팅창에 붙여넣고 "실행해줘"라고 하면 AI가 대신 실행하고 결과를 설명합니다.

### 4) 알뜰(토큰 절약) 원칙
일반 사용자는 Pro 요금제가 기본이라는 전제 아래, 스킬은 필요한 파일만 읽고 짧게 답합니다.
상시(세션마다) 토큰 비용을 **측정·관리**합니다(현재 ~1,132 tok/세션, 목표 1,500 미만 —
[docs/design/token-efficiency.md](docs/design/token-efficiency.md)).

## 정성 효과 (질적 개선)

- **진입 장벽 붕괴**: 개발이 처음인 사람도 자연어 + 복붙만으로 첫 솔루션을 화면에 띄운다.
  "배울 게 많다"는 심리적 벽 자체가 사라진다.
- **따로 놀지 않는 도구들**: 신규 솔루션끼리는 공통 통로(interface.md + 공통 함수 + 스냅샷 봉투)로
  자연스럽게 연동된다 — "A 도구 데이터를 B가 못 읽는" 사고가 구조적으로 방지된다.
- **회사 도구가 전부 같은 얼굴**: 중앙 디자인 시스템(`design/`) 하나가 단일 진실 —
  웹앱이든 파이썬이 만든 HTML 보고서든 같은 브랜드 언어를 입는다(실측으로 확인됨).
- **전체가 보인다**: 로컬 레지스트리 + `overview()`로 이 PC의 모든 솔루션 상태·버전·주고받는 데이터를
  한눈에 파악 — "누가 뭘 만들었는지 아무도 모르는" 상태에서 벗어난다.
- **안심하고 공유**: `/axfm-publish` 0단계에서 `.env`·비밀 패턴을 자동 점검해 유출 사고를 막는다.
- **재현 가능한 개발**: 결정적 스크립트 + 고정 lockfile로 "내 PC에선 됐는데"가 사라진다.
- **팀에 맞게 진화**: 디자인 교체·규약 확장·스킬 추가가 정상 사용법 — 프레임워크가 팀과 함께 자란다.

## 정량 효과 (수치로 본 개선)

> 아래 수치는 프레임워크 자체 측정 및 실측 테스트([docs/reviews/2026-07-05-field-test-macro.md](docs/reviews/2026-07-05-field-test-macro.md))
> 기준입니다. 비개발자 파일럿 실측(M3)은 로드맵의 다음 단계입니다.

| 지표 | Before (프레임워크 없이) | After (AXFM) |
|---|---|---|
| **첫 솔루션 생성** | 환경 설정·규격 협의로 수 시간~수일 | **뼈대 생성 1초 미만**, 첫 화면까지 10분 목표 안 |
| **설치 절차** | 프로젝트마다 개별 세팅 | 터미널 **2줄** (폴더 스코프 — 전역 오염 없음) |
| **실측 업무형 명령 수행** | (기준 없음) | 프레임워크 구간 **~3분**, 조사 포함 ~5분, **오류·재시도 0회** |
| **솔루션 간 연동** | 매번 규격 협의·수작업 | 공통 규약으로 `readFrom(id, name)` **1함수 호출** |
| **디자인 통일** | 도구마다 수작업, 드리프트 방치 | 토큰 기반 결정적 수출 — **UI 코드 변경 0**으로 브랜드 교체 |
| **상시 토큰 비용** | (관리 안 됨) | **~1,132 tok/세션** 측정·관리(목표 1,500 미만) |
| **자동 품질 게이트** | 없음 | **8종**(스킬 린트·스크립트 게이트·문서 게이트·E2E 2종·단위 테스트 등) |
| **공유 전 유출 점검** | 수동·누락 위험 | `publish-check` **자동**(.env/비밀 패턴 스캔) |
| **의존성 재현성** | 설치 시점마다 상이 | lockfile로 **57개 고정** — 언제 설치해도 동일 |

**요약**: 만들기(초 단위) · 연동(1함수) · 통일(코드 변경 0) · 관리(측정·게이트)의 네 축에서
**측정 가능한 비용 절감**을 확보했으며, 실전 업무형 명령 하나를 경로 구멍 0으로 통과했습니다.

## 설치 (터미널 — **작업 폴더에만 적용**)

```
claude plugin marketplace add evelynn/AXFM
cd <작업 폴더>
claude plugin install axfm@axfm --scope project
```

플러그인은 **설치한 폴더에서만 활성화**됩니다(전역 오염 없음). 다른 프로젝트에는 스킬이 나타나지 않습니다.
`/axfm-new`로 만든 솔루션 폴더에는 활성화 설정(`.claude/settings.json`)이 자동 포함됩니다.

> **터미널이 어렵다면**: 위 명령을 Claude Code 채팅창에 붙여넣고 "실행해줘"라고 하세요 —
> AI가 대신 실행하고 결과를 설명합니다.

## 시작하기

설치 후 Claude Code에서 `/axfm-new` 를 입력하세요. 10분 안에 첫 솔루션이 화면에 뜹니다.
그다음 `/axfm-guide`로 "지금 할 일"을 하나씩 안내받으세요.
자세한 안내: [docs/quickstart.md](docs/quickstart.md)

## 어떻게 연동되나 (공통 통로)

AXFM은 상시 구동 서버 없이, **문서·함수·스냅샷** 세 기둥으로 솔루션을 잇습니다.

| 기둥 | 실체 | 역할 |
|---|---|---|
| **연동 함수 문서** | 각 솔루션의 `axfm/interface.md` | "내 데이터·함수는 이렇게 쓴다"는 계약서 |
| **공통 함수 라이브러리** | vendored `lib/axfm/`(JS)·`axfm/`(Py) | 표준 읽기/쓰기 함수 — 모든 솔루션이 동일 표면 공유 |
| **연동 글루 코드** | `/axfm-connect`가 생성하는 `connectors/<상대id>.*` | 상대 문서를 읽고 연결 코드를 바이브 코딩으로 생성 |
| 로컬 레지스트리 | `~/.axfm/registry.json` | 이 PC에 어떤 솔루션이 어디 있는지 (스킬이 유지) |

발신자는 자기 폴더에 `writeShared("이름", {...})`로 스냅샷을 내보내고, 수신자는
`readFrom("발신자id", "이름")`으로 수집합니다. 상대가 실행 중이 아니어도 스냅샷 파일을 읽으므로 동작합니다.
전체 명세: [docs/protocol.md](docs/protocol.md).

## 디자인 통일

모든 AXFM 솔루션은 [design/](design/) 폴더의 디자인 시스템(DESIGN.md 규격)을 따릅니다.
디자인 md 파일은 추가/삭제할 수 있으며, 변경하면 모든 솔루션의 다음 UI 작업부터 반영됩니다.
UI는 임의 색·간격을 쓰지 않고 **디자인 토큰만** 사용하며, `export-design.mjs`가 이를 CSS 변수로
결정적으로 수출합니다. 브랜드 교체는 파일 교체/오버라이드만으로 이뤄집니다(UI 코드 변경 0).
